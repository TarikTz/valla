# weld-cli Design Spec

**Date:** 2026-03-27
**Status:** Approved

---

## Overview

`weld-cli` is an interactive, lightning-fast CLI tool built in Go that scaffolds full-stack web applications. It detects available runtimes on the user's machine, guides them through a bubbletea TUI to select their stack, runs official scaffolding commands for each piece, then wires the pieces together with CORS config, environment files, and Docker Compose.

---

## Goals

- Reduce project initialization from hours to seconds
- Use official scaffolders (not custom templates) so generated projects are always up-to-date
- Stay extensible: adding a new framework requires only a new registry YAML entry, no core changes for scaffolding (wiring patches remain framework-specific and require a registry entry update)
- Ship v1 as a binary distributed via GitHub Releases + NPX wrapper

---

## v1 Supported Matrix

| Type     | Options                     |
|----------|-----------------------------|
| Frontend | React, Vue                  |
| Backend  | Go (Gin, Fiber, Boilerplate), Node.js (Express, Nest, Boilerplate) |
| Database | PostgreSQL, SQLite           |

**8 base combinations.** Extensible to full matrix in future releases.

### Boilerplate variants

"Boilerplate" backend variants are **not** generated by an external scaffolder — weld-cli writes a minimal starter itself (`main.go` + `go.mod` for Go, `index.js` + `package.json` for Node). Registry entries for boilerplate variants set `scaffold_cmd: ""` and reference a `builtin_template` key pointing to an embedded template directory.

### Framework-specific scaffolding (Gin, Fiber, Express, Nest)

Official scaffolders (e.g., `go mod init` + `go get gin`) produce dependency manifests (`go.mod`, `package.json`) but not runnable source files. Each non-boilerplate backend registry entry therefore includes a `post_scaffold_files` block — minimal source files that weld-cli writes **after** running `scaffold_cmd`. These include a working `main.go` (Go) or `index.js` (Node) that imports the selected framework and includes a `// weld:cors` marker at the correct injection point.

The `// weld:cors` marker is **always** present because weld-cli writes it via `post_scaffold_files` (non-boilerplate) or `builtin_template` (boilerplate). It is never expected from an external scaffolder.

Example:
```yaml
post_scaffold_files:
  - path: main.go
    template: registry/post-scaffold/go-gin-main.go.tmpl
    # template includes: package main, import gin, func main(), // weld:cors marker
```

## WeldContext

All template rendering across the tool uses a single `WeldContext` struct. It is populated after step 9 (port overrides) and before any scaffolding or wiring begins.

```go
type WeldContext struct {
    ProjectName   string  // from step 1 prompt
    ScaffoldName  string  // temporary name used in scaffold_cmd (e.g., "__weld_fe__")
    FrontendPort  int     // from step 9, defaults from frontend registry entry
    BackendPort   int     // from step 9, defaults from backend registry entry
    DBPort        int     // from step 9, defaults from database registry entry (0 for SQLite)
    DBHost        string  // "localhost" (local mode) or "db" (docker mode)
    DBUser        string  // from step 9, default "postgres" / "root"
    DBPassword    string  // from step 9, default "postgres" / "root"
    DBName        string  // from step 9, defaults to ProjectName
    DBPath        string  // SQLite only, from step 9, defaults to db_path_default
    OutputMode    string  // "monorepo" or "separate"
    EnvMode       string  // "local" or "docker"
}
```

Every `{{.FieldName}}` reference in registry YAML templates, `.env` templates, and docker-compose templates maps to a field in this struct.

## Architecture

weld-cli is composed of four layers:

```
┌─────────────────────────────────────────┐
│           TUI Layer (bubbletea)         │  ← prompts, spinners, output
├─────────────────────────────────────────┤
│         Orchestrator (core CLI)         │  ← runtime detection, flow control
├─────────────────────────────────────────┤
│         Registry (YAML config)          │  ← stack definitions, defaults
├─────────────────────────────────────────┤
│      Scaffolder + Wiring Engine         │  ← runs commands, injects config
└─────────────────────────────────────────┘
```

### Registry

YAML files embedded into the binary via `go:embed`. Each entry describes one stack variant.

```yaml
# registry/backends/go-gin.yaml
id: go-gin
name: Go (Gin)
type: backend
runtime: go
group: Go
requires_runtime: go
scaffold_cmd: "go mod init {{.ScaffoldName}} && go get github.com/gin-gonic/gin"
default_port: 8080
env_vars:
  - PORT
  - DB_HOST
  - DB_PORT
  - DB_USER
  - DB_PASSWORD
  - DB_NAME
cors_patch:
  file: main.go
  marker: "// weld:cors"      # always present — written by post_scaffold_files
  template: |
    r.Use(cors.New(cors.Config{
        AllowOrigins: []string{"http://localhost:{{.FrontendPort}}"},
    }))
post_scaffold_files:
  - path: main.go
    template: registry/post-scaffold/go-gin-main.go.tmpl
docker:
  image: golang:1.22-alpine
  build_context: ./backend    # adjusted by wiring engine for Separate Folders mode
  dockerfile: registry/dockerfiles/go.Dockerfile
```

```yaml
# registry/backends/go-boilerplate.yaml
id: go-boilerplate
name: Go (Boilerplate)
type: backend
runtime: go
group: Go
requires_runtime: go
scaffold_cmd: ""
builtin_template: templates/backends/go-boilerplate
default_port: 8080
env_vars:
  - PORT
  - DB_HOST
  - DB_PORT
  - DB_USER
  - DB_PASSWORD
  - DB_NAME
cors_patch:
  file: main.go
  marker: "// weld:cors"      # always present — included in builtin_template
  template: |
    http.HandleFunc("/", corsMiddleware(handler))
docker:
  image: golang:1.22-alpine
  build_context: ./backend
  dockerfile: registry/dockerfiles/go.Dockerfile
```

```yaml
# registry/frontends/react.yaml
id: react
name: React
type: frontend
runtime: node
group: React
requires_runtime: node
scaffold_cmd: "npm create vite@latest {{.ScaffoldName}} -- --template react"
default_port: 5173
http_client_patch:
  file: src/api.ts             # weld-cli creates this file (does not exist in vanilla Vite output)
  mode: create                 # "create" = write new file; "inject" = append after marker in existing file
  template: |
    export const API_BASE = import.meta.env.VITE_API_URL ?? 'http://localhost:{{.BackendPort}}'
docker:
  image: node:20-alpine
  build_context: ./frontend
  dockerfile: registry/dockerfiles/node-vite.Dockerfile
```

```yaml
# registry/databases/postgres.yaml
id: postgres
name: PostgreSQL
type: database
default_port: 5432
docker:
  image: postgres:16-alpine
  env_vars:
    POSTGRES_USER: "{{.DBUser}}"
    POSTGRES_PASSWORD: "{{.DBPassword}}"
    POSTGRES_DB: "{{.DBName}}"
```

```yaml
# registry/databases/sqlite.yaml
id: sqlite
name: SQLite
type: database
# No default_port — SQLite is file-based
# No docker block — SQLite runs in-process, not as a container service
sqlite: true          # wiring engine omits DB service in docker-compose and uses DBPath instead of host/port vars
db_path_default: "./data/app.db"
```

Key properties:
- `group` — display grouping for two-step runtime→framework selection (steps 3/4 and 5/6)
- `requires_runtime` — binary checked by detector to filter available options
- `default_port` — pre-fills port override step; final value stored in `WeldContext`
- `scaffold_cmd` — uses `{{.ScaffoldName}}` (temporary directory name, not the final project name)
- `builtin_template` — embedded template directory for Boilerplate variants (no external scaffolder)
- `post_scaffold_files` — minimal source files written by weld-cli after `scaffold_cmd` runs
- `cors_patch.marker` — `// weld:cors` comment, always present via `post_scaffold_files` or `builtin_template`
- `http_client_patch.mode` — `create`: write new file; `inject`: append after marker in existing file
- `sqlite: true` — omit DB container from docker-compose; substitute `DBPath` for DB connection vars

## Prompt Flow

```
1. Project name          → text input
2. Output structure      → Monorepo / Separate folders
3. Frontend runtime      → filtered to installed runtimes (node detected → show React/Vue)
4. Frontend framework    → filtered to selected runtime group
5. Backend runtime       → filtered to installed runtimes
6. Backend framework     → filtered to selected runtime group (see group field)
7. Database              → PostgreSQL, SQLite
8. Environment mode      → Local (.env) / Docker Compose
9. Port overrides        → tabbed form, all defaults shown, all editable
                           (SQLite shows "DB file path" text field instead of port)
10. Confirm summary      → full review of selections before execution
```

Runtime detection runs at startup. After step 9, `WeldContext` is fully populated. All subsequent template rendering uses this context.

## Output Structure

User selects one of:

**Monorepo:**
```
my-project/
├── frontend/              # scaffolded frontend
├── backend/               # scaffolded backend
├── .env                   # always generated (local mode and docker mode)
└── docker-compose.yml     # (Docker mode only)
```

**Separate folders:**
```
my-project/
├── my-project-frontend/   # self-contained frontend project
├── my-project-backend/    # self-contained backend project
├── .env                   # always generated
└── docker-compose.yml     # (Docker mode only)
```

`.env` is generated in **both** local and Docker modes. In Docker mode it provides the variable values that `docker-compose.yml` references via `${VAR}` syntax.

## Scaffolder

Executes `scaffold_cmd` for each selected stack entry using Go's `os/exec`. Commands run in sequence with a bubbletea spinner per step.

**Directory naming reconciliation:** Official scaffolders create a subdirectory named after the argument passed (e.g., `npm create vite@latest __weld_fe__` creates `./__weld_fe__/`). weld-cli:
1. Runs `scaffold_cmd` with `{{.ScaffoldName}}` = a deterministic temporary name (e.g., `__weld_fe__`)
2. After the command succeeds, renames the output directory to the final target name:
   - Monorepo: `frontend` or `backend`
   - Separate Folders: `<project>-frontend` or `<project>-backend`

**Rollback on failure:** If any scaffold step fails, weld-cli deletes the entire project root directory and exits with an error showing the failed command's output. The boundary is always the full project root.

## Wiring Engine

Runs after all scaffold steps succeed. All templates rendered using `WeldContext`.

### 1. Environment config

`.env` is always written at the project root regardless of env mode.

**PostgreSQL:**
```
FRONTEND_PORT=5173
BACKEND_PORT=8080
DB_HOST=localhost
DB_PORT=5432
DB_NAME=myapp
DB_USER=postgres
DB_PASSWORD=postgres
VITE_API_URL=http://localhost:8080
```

**SQLite:**
```
FRONTEND_PORT=5173
BACKEND_PORT=8080
DB_PATH=./data/app.db
VITE_API_URL=http://localhost:8080
```

In Docker mode, `DB_HOST` is set to `db` (the Docker service name) instead of `localhost`, so the backend container can reach the database container.

### 2. Docker Compose (Docker mode only)

The wiring engine adjusts `build_context` values based on output structure:
- Monorepo: `./frontend`, `./backend`
- Separate Folders: `./my-project-frontend`, `./my-project-backend`

Generated structure:

```yaml
version: "3.8"
networks:
  weld-net:
    driver: bridge

services:
  frontend:
    build:
      context: <adjusted build_context>
      dockerfile: <registry docker.dockerfile>
    ports:
      - "5173:5173"
    environment:
      - VITE_API_URL=http://backend:8080   # uses service name, not localhost
    networks: [weld-net]
    depends_on: [backend]

  backend:
    build:
      context: <adjusted build_context>
      dockerfile: <registry docker.dockerfile>
    ports:
      - "8080:8080"
    env_file: .env
    networks: [weld-net]
    depends_on: [db]     # omitted if SQLite

  db:                    # omitted if SQLite
    image: postgres:16-alpine
    ports:
      - "5432:5432"
    environment:
      POSTGRES_USER: ${DB_USER}
      POSTGRES_PASSWORD: ${DB_PASSWORD}
      POSTGRES_DB: ${DB_NAME}
    volumes:
      - db-data:/var/lib/postgresql/data
    networks: [weld-net]

volumes:
  db-data:               # omitted if SQLite
```

### 3. CORS patch

Finds `// weld:cors` in the generated backend file (`cors_patch.file`) and injects the rendered template on the immediately following line. The marker is guaranteed to be present (written by `post_scaffold_files` or `builtin_template`). If the marker is not found (defensive case), weld-cli prints a warning but does not fail.

### 4. HTTP client patch

For `mode: create` — weld-cli writes a new file at the specified path within the frontend directory. For `mode: inject` — finds a marker in an existing file and injects after it. React and Vue use `mode: create` since `src/api.ts` does not exist in vanilla Vite output.

## Distribution (v1)

1. **GoReleaser** compiles binaries for macOS (ARM/Intel), Linux (x64/ARM), Windows (x64) and attaches them to GitHub Releases
2. **NPX wrapper** (`npm/index.js`):
   - Version: read from `npm/package.json` `version` field — stored without `v` prefix (e.g., `"1.0.0"`)
   - GitHub Release tag uses `v` prefix: `v1.0.0`; archive filename does not: `weld-cli_1.0.0_Linux_x86_64.tar.gz`
   - URL pattern: `https://github.com/<org>/weld-cli/releases/download/v{version}/weld-cli_{version}_{OS}_{Arch}.tar.gz`
   - OS mapping: `process.platform` → `darwin`→`Darwin`, `linux`→`Linux`, `win32`→`Windows`
   - Arch mapping: `process.arch` → `x64`→`x86_64`, `arm64`→`arm64`
   - Archive format: `.tar.gz` (Unix), `.zip` (Windows)
   - Extraction: Node built-in `zlib` + `tar` module (no external dependencies)
   - Cache path (Unix): `~/.cache/weld-cli/{version}/weld-cli`
   - Cache path (Windows): `%LOCALAPPDATA%\weld-cli\{version}\weld-cli.exe`
   - Cache hit: version directory exists with executable → skip download
   - Exec: `child_process.execFileSync(binaryPath, process.argv.slice(2), { stdio: 'inherit' })`
3. User runs: `npx weld-cli`

Future: Homebrew tap, Chocolatey, `.deb`/`.rpm` packages.

## Extensibility

Adding a new framework to weld-cli:
1. Add a YAML file to `registry/backends/` (or `frontends/`, `databases/`)
2. Add a `cors_patch` or `http_client_patch` entry with appropriate marker and template
3. Add a `post_scaffold_files` template (for non-boilerplate backends) to `registry/post-scaffold/`

The core orchestration, TUI, and scaffolder require no changes.

## Non-Goals (v1)

- Full 4×5×4 matrix support (v2+)
- Homebrew / Chocolatey / Linux packages (v2+)
- Authentication scaffolding
- CI/CD pipeline generation
- Cloud deployment config
